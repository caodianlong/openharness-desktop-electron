# 产品需求文档 PRD v0.1

## 1. 产品定义

### 1.1 产品定位
打造一个以 **OpenHarness** 为执行核心、提供完整桌面工作台体验的 AI Agent 桌面应用。产品目标接近 Claude Cowork 的桌面体验，但底层不依赖 CLI，而是将 OpenHarness 作为 **Python 库模块** 深度嵌入。

### 1.2 产品目标
- 让用户以“桌面应用”的方式使用 OpenHarness，而不是命令行工具
- 提供完整的：
  - 会话管理
  - 会话历史
  - 任务运行
  - 审批控制
  - 文件/产物预览
  - 设置与恢复
- 默认安全模式，但必须支持 **Full Auto**
- 后端对用户不可见：
  - 一体启动
  - 一体退出
  - 无命令行黑窗口
- 保持 OpenHarness 可持续升级，桌面壳不与其私有实现强耦合

### 1.3 成功标准
- 冷启动进入主界面：**≤ 2.5 秒**
- 应用启动后 Host ready：**≤ 4 秒**
- 会话切换响应：**≤ 200ms**
- 新建会话到可发送第一条消息：**≤ 1 秒**
- 崩溃重启后可恢复最近会话与未完成 run
- OpenHarness 小版本升级不要求重构 UI 层

## 2. 用户画像与场景

### 2.1 核心用户画像

| 用户类型 | 特征 | 核心诉求 |
|---|---|---|
| 个人开发者 | 高频使用 AI 做项目任务 | 一体化、稳定、可追溯 |
| 技术团队成员 | 多任务并行，需要结果和过程可审计 | 任务流、日志、审批 |
| AI 重度用户 | 长会话、多阶段执行 | 历史管理、恢复、Full Auto |
| 产品/运营协作者 | 不关心命令行 | 看结果、看文件、做审批 |

### 2.2 典型使用场景
1. **持续型项目会话**：围绕一个 workspace 连续工作数天，要求历史、运行、产物完整保留。
2. **审批驱动的自动化执行**：默认模式下敏感动作需审批，支持按作用域授权。
3. **Full Auto 模式**：用户显式开启后，在限定范围内自动执行，要求可见、可中断、可审计。
4. **产物交付与查看**：在应用内查看文档、代码、diff、日志、图片、PDF。
5. **异常恢复**：Host 或 UI 崩溃后恢复最近会话与运行状态。

## 3. 核心问题与产品价值

### 3.1 当前问题
- CLI 交互不等于桌面产品体验
- 进程与运行状态对普通用户不透明
- 会话、任务、审批、文件、日志分散
- 历史和恢复能力不足
- 若依赖 CLI 集成，上游升级会持续破坏产品层

### 3.2 产品价值
- **统一体验**：一个桌面应用，屏蔽底层复杂度
- **可控自动化**：默认安全，支持 Full Auto
- **可追溯**：会话、审批、产物、日志、trace 全链路
- **可恢复**：崩溃后可回到工作现场
- **可升级**：OpenHarness 升级收敛在适配层

## 4. 功能范围

### 4.1 MVP

#### 必须包含
- 桌面应用壳
- 会话列表：新建、重命名、归档、删除
- 会话消息流
- 基础 run / task 状态展示
- 审批弹窗与审批队列
- 文件/产物列表
- 基础预览器：文本、Markdown、图片、PDF、JSON
- 设置中心：模型参数、默认工作区、权限模式、日志与诊断
- 本地数据库（SQLite）
- 本地归档（artifacts / logs / snapshots）
- Host 启停与健康检查
- 崩溃恢复
- Safe / Balanced / Full Auto 模式切换

#### 明确不做
- 多人协作
- 云同步
- 插件市场
- 企业后台
- 复杂多窗口协同
- 重写 OpenHarness 内核

### 4.2 Beta
- 任务中心
- 工作区管理
- 审批规则模板
- 更丰富预览器（diff、代码高亮、HTML sandbox）
- trace 浏览器
- 会话搜索
- 历史筛选与标签
- OpenHarness 版本管理
- 增强恢复（run 级恢复）

### 4.3 1.0
- 多项目工作区
- 细粒度权限策略
- 任务看板
- 批量审批
- 完整诊断报告
- 自动更新与回滚
- 企业策略注入预留
- 远程执行扩展预留

## 5. 非功能需求

### 5.1 性能
- 冷启动：≤ 2.5 秒
- Host ready：≤ 4 秒
- 会话切换：≤ 200ms
- 审批 UI 响应：≤ 300ms
- 大会话首屏渲染：≤ 800ms

### 5.2 稳定性
- Host 崩溃不能直接拖死 UI
- UI 崩溃后重启可恢复最近工作上下文
- 关键写操作必须事务化
- 关键事件必须落盘

### 5.3 可恢复性
- 恢复最近 conversation / run / pending approvals
- 检测异常退出标志
- 支持查看上次崩溃摘要

### 5.4 权限与安全
- 默认非 Full Auto
- 敏感操作需审批或预授权
- 权限模型必须可见、可切换、可审计
- 访问范围可控
- 审批记录可追溯

### 5.5 诊断与追溯
- 每个 run 有 trace_id
- 工具调用、审批、产物、错误全链路关联
- 支持导出诊断包

### 5.6 升级兼容
- UI 不直接依赖 OpenHarness 私有内部结构
- OpenHarness 升级收敛在 Host Adapter
- 升级失败必须可回退

## 6. 关键交互说明

### 6.1 首次启动
1. 用户启动桌面应用
2. Electron Main 拉起 Python Headless Host
3. UI 显示启动状态
4. Host ready 后进入主界面
5. 首次使用引导设置工作区、模型、权限模式

### 6.2 新建会话
1. 用户点击“新建会话”
2. 选择工作区与权限模式
3. 创建 conversation
4. 初始化 runtime context
5. 进入主工作区界面

### 6.3 执行与审批
1. 用户发送任务
2. OpenHarness 运行
3. 如触发敏感 action，生成 approval request
4. UI 弹出审批卡片或放入审批队列
5. 用户批准/拒绝/按作用域授权
6. Host 接收决策后继续执行

### 6.4 Full Auto 切换
1. 用户显式开启
2. 系统展示风险说明与作用域
3. 记录审计事件
4. 顶部常驻 Full Auto 标识
5. 支持随时关闭并回退为审批模式

### 6.5 崩溃恢复
1. 启动时检测异常退出
2. 恢复最近快照
3. 标记未完成 run
4. 用户可继续/终止/查看日志

## 7. OpenHarness 与桌面壳职责边界

### 7.1 依赖 OpenHarness 的能力
- Agent 执行核心
- 工具编排
- 任务/子代理运行
- 行为事件流
- action / approval 触发
- artifact 输出元数据
- 运行时上下文管理

### 7.2 由桌面壳负责的能力
- 桌面生命周期
- Host 托管
- 本地数据库
- 文件归档与索引
- 会话列表与历史浏览
- 审批 UI 与规则表达
- 设置中心
- 诊断与日志导出
- 崩溃恢复编排
- 安装/打包/更新
