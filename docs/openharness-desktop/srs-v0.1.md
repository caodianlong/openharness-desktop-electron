## 1. 引言

### 1.1 编写目的
本文档用于将《OpenHarness Desktop Electron PRD v0.2》转化为可直接用于研发排期、系统设计、测试用例编写和验收评审的**软件需求规格说明书（SRS）**。

本文档的目标是：
- 明确产品边界、角色、核心流程与约束条件；
- 将模糊产品诉求转化为结构化、可验证、可测试的需求条目；
- 为前端、Host、适配层、数据层、测试与运维提供统一基线；
- 为后续 Beta/1.0 规划保留可演进空间。

### 1.2 产品范围
OpenHarness Desktop 是一个面向通用知识工作场景的桌面 AI 协作助手，采用 Electron 作为桌面壳，以 OpenHarness 作为内嵌执行核心，向用户提供以下闭环能力：
- 创建并持续推进任务型会话；
- 发起 Agent 执行并观察执行过程；
- 对敏感操作进行审批或预授权；
- 管理输入材料、运行上下文与输出产物；
- 在崩溃或重启后恢复工作现场；
- 在单一桌面工作台内完成搜索、阅读、写作、文件处理、数据分析与自动化执行。

### 1.3 非目标（Out of Scope）
本版本不包含以下能力：
- 多人实时协作；
- 云同步与跨设备会话同步；
- 插件市场；
- 企业管理后台；
- 重写 OpenHarness 内核；
- 多窗口复杂协同工作流；
- 第三方 IM / 文档 / 任务系统的正式集成能力。

### 1.4 定义与术语

| 术语 | 定义 |
|---|---|
| OpenHarness | 作为桌面应用内嵌执行核心的 Agent 运行时与工具编排系统 |
| Desktop Shell | 基于 Electron 的桌面外壳，负责 UI、生命周期、数据库、本地归档、恢复、审批界面等 |
| Host | 以 import 方式集成 OpenHarness 的本地后端进程，负责运行会话、桥接事件、管理运行时 |
| Conversation / 会话 | 围绕一个目标持续协作的主容器，包含消息、输入、运行、产物与摘要 |
| Run | 用户一次任务提交后触发的一次执行周期 |
| Approval | 对敏感动作的用户审批请求 |
| Artifact / 产物 | Run 过程中生成的文件、日志、结构化结果、报告、代码、图片等 |
| Workspace | 用户任务的默认工作目录或文件上下文范围 |
| Safe / Balanced / Full Auto | 三种自动化权限模式，控制系统对敏感动作的默认处理方式 |
| Host Adapter | 桌面壳与 OpenHarness 之间的适配层，负责隔离上游实现差异 |
| Trace | 一次 Run 的全链路追踪标识与关联信息 |

## 2. 系统概述

### 2.1 产品定位
OpenHarness Desktop 是一个**通用办公桌面 AI 协作助手**，不是单次问答工具，也不是仅面向开发者的编码壳。产品强调：
- 长时段协作；
- 任务闭环；
- 可控自动化；
- 过程透明；
- 崩溃可恢复；
- 用户无需理解 CLI、日志结构和进程细节。

### 2.2 产品核心价值
1. **统一工作台**：输入、执行、查看、导出、继续推进全部在一个应用内完成；
2. **默认安全**：敏感动作默认经过审批或预授权；
3. **透明执行**：用户能看到工具调用、产物、日志、错误与当前阶段；
4. **可恢复协作**：中断后回到最近工作现场；
5. **升级可维护**：OpenHarness 升级被收敛在 Host Adapter，不要求 UI 重构。

### 2.3 用户模型
系统覆盖以下 5 类主要用户：
- 知识工作者；
- 产品/项目管理者；
- 运营人员；
- 开发者；
- AI 重度用户。

### 2.4 运行环境

#### 2.4.1 客户端环境
- 操作系统：macOS / Windows / Linux 桌面环境（MVP 以桌面端为唯一正式交付平台）
- 应用形态：Electron 单窗口应用
- 网络要求：支持离线基础能力，联网增强执行能力

#### 2.4.2 后端运行环境
- 本地 Host 进程
- 本地 SQLite 数据库
- 本地归档目录（artifacts / logs / snapshots）
- OpenHarness 运行时通过 import 集成，不经 CLI 包装

#### 2.4.3 技术架构基线
- UI：Electron + Web 前端 SPA
- Host：Python 本地服务（HTTP + WebSocket）
- 持久化：SQLite + JSON 快照双层存储
- 通信：桌面壳与 Host 通过本地 HTTP/WebSocket 协议交互

### 2.5 架构原则
1. **桌面壳负责产品态，OpenHarness 负责执行态**；
2. **前后端职责边界清晰**，UI 不直连 OpenHarness 私有内部结构；
3. **同一任务支持多阶段推进**，会话是一级对象，Run 是会话下的执行对象；
4. **所有关键事件可审计、可恢复、可导出**；
5. **高频主路径优先**：先保证长会话协作闭环，再扩展组织级能力。

### 3.1 用户角色定义

| UR-02 | 产品/项目管理 | 拆解任务、推进事项、查看阶段结果、做审批 | 状态可见、阶段清晰、可恢复、可交付 |

| UR-04 | 开发者 | 代码处理、脚本执行、日志排查、自动化操作 | 运行稳定、日志清晰、权限可控、可追溯 |

### 3.2 核心用户故事总览

1. **As a 知识工作者, I want** 将文件、链接和说明一起带入会话，**so that** AI 能基于完整上下文持续帮我分析和产出。
2. **As a 产品/项目管理者, I want** 在一个长期会话中看到阶段进展、审批请求和最近结果，**so that** 我能持续推进任务而不丢失上下文。
3. **As a 运营人员, I want** 在一个工作台里完成搜索、整理、写作与导出，**so that** 我不用在多个工具之间来回切换。

## 4. 功能性需求

> 说明：
> - FR 编号全局连续，无遗漏；
> - 优先级定义：P0=首发必需，P1=首发重要，P2=可延后；
> - 每条需求均包含：编号、名称、优先级、用户故事、描述、业务规则、边界条件、验收标准；
> - 标注“已验证可行”表示该能力已有技术验证或主干链路验证结果支持。

### 4.1 模块 A：桌面应用启动与 Host 生命周期管理

#### FR-001 应用冷启动与主界面进入
- **优先级**：P0
- **用户故事**：As a 通用用户, I want 应用启动后快速进入主界面, so that 我能立即开始工作。
- **描述**：系统应在用户启动应用后，展示启动状态并在 Host ready 后进入主界面。
- **业务规则**：
  1. 启动过程必须包含“应用启动中”“Host 启动中”“准备完成”三个可识别状态；
  2. 当 Host ready 前，主输入区不可发送任务；
  3. 若启动异常，必须显示错误信息和重试入口。
- **边界条件**：
  - 首次启动无历史数据；
  - Host 启动时间超出阈值；
  - 本地数据库损坏或不可访问。
- **验收标准**：
  1. 冷启动进入主界面时间 ≤ 2.5 秒；
  2. Host ready 时间 ≤ 4 秒；
  3. Host 未就绪时不出现可误用的输入可发状态；
  4. 启动失败时可见错误文案、日志摘要及重试按钮。

#### FR-002 Host 后台托管与无黑窗运行
- **优先级**：P0
- **用户故事**：As a 非技术用户, I want 不看到底层进程和命令行窗口, so that 产品更像正常桌面软件。
- **描述**：Electron Main 必须负责拉起、托管并关闭 Host 进程，对用户隐藏底层进程细节。
- **业务规则**：
  1. Host 必须以后台方式运行；
  2. 退出应用时默认同时结束 Host；
  3. Host 异常退出不得直接拖死 UI。
- **边界条件**：
  - Host 进程异常退出；
  - UI 正常退出但 Host 未响应；
  - 重复启动应用实例。
- **验收标准**：
  1. 启动与退出过程无命令行黑窗口暴露；
  2. Host 崩溃后 UI 仍可展示故障态；
  3. 关闭应用后不残留孤儿 Host 进程。

#### FR-003 Host 健康检查与故障提示
- **优先级**：P0
- **用户故事**：As a 用户, I want 知道系统是否处于可执行状态, so that 我不会在不可用状态下误发任务。
- **描述**：系统必须持续检查 Host 健康状态，并将结果可视化展示。
- **业务规则**：
  1. 至少检测连接可达、协议可用、运行时可提交任务三个维度；
  2. 状态分为：正常、启动中、退化、不可用；
  3. 状态变化必须有时间戳记录。
- **边界条件**：
  - 短时网络闪断；
  - Host 局部接口异常；
  - 数据库可用但 WS 连接失败。
- **验收标准**：
  1. Host 异常后 5 秒内 UI 状态更新；
  2. 不同故障类型显示差异化提示；
  3. 用户可从故障态进入诊断或重试流程。

#### FR-004 单实例启动约束
- **优先级**：P1
- **用户故事**：As a 用户, I want 避免多个应用实例抢占同一数据源, so that 会话与 Host 状态保持一致。
- **描述**：系统应默认限制同一用户环境下同一版本桌面应用仅允许单实例主运行。
- **业务规则**：
  1. 若检测到已有实例，后启动实例应聚焦到已有窗口；
  2. 不允许两个主实例同时写同一数据库；
  3. 若用户显式选择“只读诊断模式”，可作为例外机制在 Beta 再扩展。
- **边界条件**：
  - 主实例卡死；
  - 旧进程残留锁；
  - 系统崩溃后重启。
- **验收标准**：
  1. 重复打开应用时不产生双写；
  2. 冲突时给出可理解提示；
  3. 异常锁可在恢复流程中清理。

### 4.2 模块 B：会话管理

#### FR-005 新建会话
- **优先级**：P0
- **用户故事**：As a 用户, I want 快速创建一个新任务会话, so that 我能围绕单一目标持续协作。
- **描述**：用户可新建会话，并在创建时指定工作区和权限模式。
- **业务规则**：
  1. 新会话创建后应立即进入可输入状态；
  2. 未指定工作区时使用默认工作区；
  3. 会话初始标题应为“未命名会话”或占位标题，后续可自动生成。
- **边界条件**：
  - 默认工作区不存在；
  - 权限模式非法；
  - 新建后 Host 尚未 ready。
- **验收标准**：
  1. 新建会话到可发送第一条消息 ≤ 1 秒；
  2. 新建后自动切换到新会话；
  3. 会话具备唯一 ID，但 UI 默认展示标题而非 ID。

#### FR-006 会话列表浏览与切换
- **优先级**：P0
- **用户故事**：As an AI 重度用户, I want 快速在多个长期任务之间切换, so that 我能管理并行工作。
- **描述**：系统应提供侧边栏会话列表，支持浏览、排序、切换和恢复。
- **业务规则**：
  1. 列表项展示标题、最近更新时间、状态摘要；
  2. 当前会话须高亮；
  3. 切换会话时应加载其消息、输入、产物和最近上下文摘要。
- **边界条件**：
  - 会话数量很大；
  - 当前会话存在未提交输入；
  - 切换目标会话数据不完整。
- **验收标准**：
  1. 会话切换响应 ≤ 200ms；
  2. 切换后主视图与会话一致；
  3. 列表中不可仅显示会话 ID 充当标题。
- **已验证可行**：会话 CRUD + 搜索 + 自动标题主干已完成技术验证。

#### FR-007 会话重命名
- **优先级**：P0
- **用户故事**：As a 产品/项目管理者, I want 给会话起清晰标题, so that 我能长期管理任务。
- **描述**：用户可手动重命名会话，系统也可自动生成标题。
- **业务规则**：
  1. 手动标题优先级高于自动标题；
  2. 自动标题默认从首条用户输入规则提取；
  3. 自动标题生成后，用户仍可再次修改。
- **边界条件**：
  - 首条消息为空或仅 slash 命令；
  - 标题过长；
  - 标题含非法文件字符。
- **验收标准**：
  1. 用户修改后列表与详情实时更新；
  2. 自动标题应在首轮交互结束后生成；
  3. 标题长度超限时自动截断并保留可读性。
- **已验证可行**：规则提取自动标题已验证。

#### FR-008 会话归档与删除
- **优先级**：P0
- **用户故事**：As a 用户, I want 清理历史会话但不轻易丢失数据, so that 我的工作台保持整洁又安全。
- **描述**：系统需同时支持归档与删除，优先使用可恢复的归档机制。
- **业务规则**：
  1. 默认提供“归档”而非直接永久删除；
  2. 永久删除必须二次确认；
  3. 删除会话时应同步处理其关联消息、摘要、产物索引和本地快照引用。
- **边界条件**：
  - 会话存在未完成 Run；
  - 产物文件仍被其他会话引用；
  - 数据库删除成功但文件删除失败。
- **验收标准**：
  1. 归档会话不出现在默认列表中；
  2. 删除操作有明确确认和结果反馈；
  3. 删除失败时给出局部失败说明并保留可修复状态。

#### FR-009 会话搜索
- **优先级**：P1
- **用户故事**：As an AI 重度用户, I want 通过标题或内容快速找到历史会话, so that 我能恢复长期任务。
- **描述**：系统应支持基于标题、最近消息和标签摘要的会话搜索。
- **业务规则**：
  1. MVP 至少支持标题和最近消息关键词搜索；
  2. 搜索结果按相关度和最近更新时间排序；
  3. 搜索命中字段应高亮。
- **边界条件**：
  - 大量历史记录；
  - 中英文混合搜索；
  - 已归档会话是否纳入搜索。
- **验收标准**：
  1. 关键词搜索结果在 500ms 内返回；
  2. 命中结果可直接打开对应会话；
  3. 无结果时提供空状态提示。
- **已验证可行**：会话搜索主干已完成技术验证。

### 4.3 模块 C：消息流与任务提交

#### FR-010 消息输入与发送
- **优先级**：P0
- **用户故事**：As a 用户, I want 用自然语言提交任务, so that 我能像与同事协作一样推动工作。
- **描述**：系统应提供稳定的消息输入区，支持文本输入、发送、停止和再次编辑后重发。
- **业务规则**：
  1. Host 未就绪时输入区可编辑但不可发送；
  2. 发送期间按钮切换为“停止”；
  3. 输入框应保留未发送草稿，除非用户主动清空。
- **边界条件**：
  - 空消息；
  - 超长消息；
  - 执行中重复点击发送。
- **验收标准**：
  1. 非空消息可正常发起 Run；
  2. 执行中可点击停止并收到反馈；

#### FR-011 流式回复展示
- **优先级**：P0

- **描述**：系统应支持流式展示助手回复增量，并在完成后归并为完整消息。
- **业务规则**：
  1. 每个 assistant run 的文本增量必须附着到当前运行中的响应卡片；
  2. 完成时应切换为稳定消息态；
  3. 中途中断需保留已生成内容并标注状态。
- **边界条件**：
  - 流式事件乱序；
  - 生成中断；
  - 完成事件缺失。
- **验收标准**：
  1. 文本增量连续显示，无明显闪烁或覆盖错误；
  2. Run 完成后消息定格为最终内容；
  3. 中断后用户能看懂是“完成”“失败”还是“已停止”。
- **已验证可行**：WebSocket 实时消息流已验证。

#### FR-012 工具调用卡片展示
- **优先级**：P0
- **用户故事**：As a 开发者或高频用户, I want 看到系统调用了哪些工具, so that 我能判断执行过程是否可信。
- **描述**：系统需将工具开始、进行中、完成、失败状态以可折叠卡片方式展示。
- **业务规则**：

  2. 同一 Run 下工具事件需按时序关联；

- **边界条件**：

  - 工具失败缺少堆栈信息。
- **验收标准**：
  1. `tool.started` 与 `tool.completed` 正确聚合到同一卡片；
  2. 用户可展开查看细节；
  3. 工具失败时卡片有明显失败态样式与错误摘要。
- **已验证可行**：工具执行卡片主干链路已验证。

#### FR-013 系统消息与错误消息展示
- **优先级**：P0

- **描述**：系统需在消息流中展示系统状态、故障提示、审批请求结果等非聊天类消息。
- **业务规则**：

### 4.4 模块 D：运行状态与过程控制

#### FR-015 Run 状态可视化
- **优先级**：P0
- **用户故事**：As a 产品/项目管理者, I want 知道当前任务处于哪个阶段, so that 我能判断是否等待、干预或继续。
- **描述**：系统应展示 Run 的生命周期状态，包括排队、执行中、等待审批、完成、失败、已停止、待恢复。
- **业务规则**：
  1. 同一时刻一个会话可有 0 或 1 个前台活跃 Run；
  2. 状态变更必须带时间戳；
  3. 状态与消息流、审批区、产物区同步更新。
- **边界条件**：
  - 状态回跳；
  - UI 重连；
  - Host 重启后状态不一致。
- **验收标准**：
  1. Run 状态可在主界面清楚识别；
  2. 不同状态视觉标识清晰；
  3. 状态变更具备事件记录。

#### FR-016 停止运行
- **优先级**：P0
- **用户故事**：As a 用户, I want 随时中止系统执行, so that 我能在风险或无效执行时立刻收回控制权。
- **描述**：执行期间用户可主动停止当前 Run。
- **业务规则**：
  1. 停止请求优先级高于普通后续事件；
  2. 停止后系统保留已产生内容与产物；
  3. 停止不会删除历史记录。
- **边界条件**：
  - 正在等待工具返回；
  - 正在等待审批；
  - 停止请求与完成事件同时到达。
- **验收标准**：
  1. 用户发起停止后 3 秒内状态变更可见；
  2. 已停止 Run 不再继续生成新内容；
  3. 后续可基于当前状态继续执行。

#### FR-017 Run 完成摘要与下一步推荐
- **优先级**：P1
- **用户故事**：As a 知识工作者, I want 在一次执行结束后快速理解结果并继续推进, so that 我能减少重新阅读成本。
- **描述**：Run 结束后系统应提供简要完成摘要及可继续动作建议。
- **业务规则**：
  1. 摘要需覆盖：本次做了什么、生成了什么、是否有待处理事项；
  2. 推荐动作包括继续追问、基于产物继续、导出结果等；
  3. 摘要来源可为系统生成或模型产出，但必须可回看。
- **边界条件**：
  - 结果为空；
  - 多产物输出；
  - 失败结束。
- **验收标准**：
  1. 完成态至少显示一个结果摘要区域；
  2. 用户可直接从摘要进入下一步操作；
  3. 失败结束时摘要转为问题总结与修复建议。

#### FR-018 运行轨迹与 trace_id 关联
- **优先级**：P1
- **用户故事**：As a 开发者, I want 每次执行都能关联 trace, so that 我能做诊断和审计。
- **描述**：系统必须为每个 Run 分配 trace_id，并将工具调用、审批、产物、错误关联到该 trace。
- **业务规则**：
  1. trace_id 必须全局唯一；
  2. trace 与 conversation_id、run_id 形成可追溯关联；
  3. 导出诊断包时必须带 trace 信息。
- **边界条件**：
  - Run 恢复后 trace 连续性；
  - 子代理嵌套调用；
  - 部分事件缺失 trace。
- **验收标准**：
  1. 单次 Run 的关键信息可按 trace 汇总；
  2. 导出后可离线分析；
  3. trace 缺失时系统记录数据不完整警告。

### 4.5 模块 E：审批与权限模式

#### FR-019 Safe / Balanced / Full Auto 模式切换
- **优先级**：P0
- **用户故事**：As a 用户, I want 根据任务风险切换自动化级别, so that 我能平衡效率和安全。
- **描述**：系统必须支持 Safe、Balanced、Full Auto 三种权限模式。
- **业务规则**：
  1. 默认模式为 Safe 或 Balanced，不得默认进入 Full Auto；
  2. 三种模式需显示清晰说明；
  3. 模式切换必须记录审计事件。
- **边界条件**：
  - 执行中切换；
  - 恢复会话后模式不一致；
  - 用户误触 Full Auto。
- **验收标准**：
  1. 用户可在设置或当前会话中切换模式；
  2. Full Auto 必须有二次确认；
  3. 当前模式在会话界面持续可见。

#### FR-020 敏感动作审批请求
- **优先级**：P0
- **用户故事**：As a 用户, I want 敏感动作在执行前被拦下并解释原因, so that 我能掌控风险。
- **描述**：系统在命中敏感动作时，必须生成审批请求并暂停执行，等待用户决策。
- **业务规则**：
  1. 审批卡片至少包含：动作类型、目标对象、影响范围、建议理由；
  2. 未决审批期间 Run 状态为“等待审批”；
  3. 审批决定包括：批准一次、拒绝、按作用域授权。
- **边界条件**：
  - 多个审批同时出现；
  - 审批超时；
  - UI 重启后存在未处理审批。
- **验收标准**：
  1. 审批出现后 1 秒内用户可见；
  2. 用户决策后 Host 能继续或终止执行；
  3. 审批请求具备历史追溯能力。

#### FR-021 审批队列与批量处理入口
- **优先级**：P1
- **用户故事**：As a 产品/项目管理者, I want 统一看到待审批项, so that 我能集中处理阻塞。
- **描述**：系统应提供审批队列视图，展示当前会话及全局待处理审批。
- **业务规则**：
  1. 队列按时间排序；
  2. 列表展示会话标题、审批类型、等待时长；
  3. MVP 支持单条处理，批量审批为 1.0 预留。
- **边界条件**：
  - 审批项已失效；
  - 会话被删除；
  - Full Auto 下无审批项。
- **验收标准**：
  1. 待审批数量在主界面可见；
  2. 点击队列项可定位对应会话；
  3. 处理后队列状态即时更新。

#### FR-022 作用域授权
- **优先级**：P1
- **用户故事**：As an AI 重度用户, I want 对某一类动作做有限授权, so that 我能减少重复审批。
- **描述**：系统应支持按会话、路径、工具类型等范围进行临时预授权。
- **业务规则**：
  1. 作用域必须可见、可撤销、可到期；
  2. 授权不得默认无限期；
  3. Full Auto 视为最宽作用域授权，但仍可被用户即时关闭。
- **边界条件**：
  - 同一动作命中多个作用域；
  - 授权规则冲突；
  - 恢复后作用域失效。
- **验收标准**：
  1. 用户可查看当前会话已有授权；
  2. 命中授权的动作不再弹重复审批；
  3. 超出范围时仍触发审批。

#### FR-023 Full Auto 风险提示与常驻标识
- **优先级**：P0
- **用户故事**：As a 用户, I want 在 Full Auto 下持续被提醒当前风险状态, so that 我不会忘记系统处于自动执行模式。
- **描述**：当开启 Full Auto 后，系统需展示风险说明并在主界面常驻标识。
- **业务规则**：
  1. 首次开启必须弹出风险说明；
  2. 顶部或显著区域常驻“Full Auto”状态标签；
  3. 用户可随时一键关闭并退回审批模式。
- **边界条件**：
  - 应用重启后恢复到 Full Auto；
  - 不同会话模式不一致；
  - 用户忽略风险弹窗。
- **验收标准**：
  1. Full Auto 启用时主界面显著可见；
  2. 关闭操作即时生效；
  3. 全过程有审计记录。

### 4.6 模块 F：输入材料与上下文管理

- **验收标准**：

  - 超大文件；

- **验收标准**：
  1. 用户可在发送前查看已附加文件；
  2. 文件状态可见且可移除；
  3. Run 完成后能追溯某个产物是否来源于该输入文件。

#### FR-026 URL 粘贴与引用状态识别
- **优先级**：P1
- **用户故事**：As a 运营人员, I want 粘贴链接后系统识别它是可处理材料, so that 我不用手动解释太多。
- **描述**：系统应将 URL 识别为独立输入对象，而非普通文本噪声。
- **业务规则**：
  1. URL 被识别后显示引用卡片；
  2. 用户可补充“如何使用该 URL”；
  3. 若系统支持抓取/解析，应在执行前标记其是否纳入上下文。
- **边界条件**：
  - 非法 URL；
  - 抓取失败；
  - 同时粘贴多个 URL。
- **验收标准**：
  1. URL 与普通文本在 UI 上有区分；
  2. 解析失败时有清晰降级提示；
  3. 历史中能回看引用对象及其处理结果。

#### FR-027 工作区文件或目录引用
- **优先级**：P1
- **用户故事**：As a 开发者或高频用户, I want 引用整个工作区中的文件或目录, so that AI 能在受控范围内读取上下文。
- **描述**：用户可从当前工作区中引用文件或目录作为任务上下文范围。
- **业务规则**：
  1. 系统必须明确展示引用范围；
  2. 引用范围受当前权限模式约束；
  3. 目录引用应提示潜在访问规模。
- **边界条件**：
  - 目录过大；
  - 无权限访问；
  - 引用后路径失效。
- **验收标准**：
  1. 用户能清楚看到引用的是文件还是目录；
  2. 超大范围引用前给出提示；
  3. 引用对象被纳入审计记录。

#### FR-028 历史产物引用继续执行
- **优先级**：P0
- **用户故事**：As an AI 重度用户, I want 基于已有产物继续下一步任务, so that 我不用重新组织上下文。
- **描述**：用户可从产物列表或预览器中选择一个已有产物，基于其发起下一步 Run。
- **业务规则**：
  1. 系统自动生成带产物引用的上下文；
  2. 产物与新 Run 之间建立父子关系；
  3. 历史中可回看“由哪个产物继续而来”。
- **边界条件**：
  - 产物已被删除；
  - 多个产物一起引用；
  - 原产物格式不兼容。
- **验收标准**：
  1. UI 中存在“基于此继续”入口；
  2. 新 Run 可识别到所引用产物；
  3. 用户无需手工复制全文即可推进任务。

#### FR-029 上下文生命周期管理
- **优先级**：P0
- **用户故事**：As an AI 重度用户, I want 系统区分即时输入、摘要和持久引用, so that 长会话不会失控。
- **描述**：系统必须管理上下文层级，包括即时输入、阶段摘要、历史折叠内容、持久引用对象。
- **业务规则**：
  1. 折叠和摘要不等于删除；
  2. 当前有效上下文需有显式表示；
  3. 恢复时优先加载当前有效上下文而非全部原始内容。
- **边界条件**：
  - 会话极长；
  - 摘要失败；
  - 恢复时上下文不一致。
- **验收标准**：
  1. 用户能区分当前有效上下文与历史折叠内容；
  2. 长会话仍可继续执行；
  3. 恢复后默认展示的是可工作的当前上下文视图。

#### FR-031 基础预览器
- **优先级**：P0
- **用户故事**：As a 用户, I want 直接在应用内查看常见结果文件, so that 我不用反复跳外部应用。
- **描述**：MVP 必须支持文本、Markdown、图片、PDF、JSON 的内嵌预览。
- **业务规则**：
  1. 预览器失败时降级为基础文件信息视图；
  2. 超大文件可采用按需加载；
  3. 预览状态不得阻塞主界面可交互性。
- **边界条件**：
  - 文件编码异常；
  - PDF 页数很多；
  - JSON 体积过大。
- **验收标准**：
  1. 5 类基础格式均可打开；
  2. 预览失败时用户仍可看到文件元信息；
  3. 打开预览不会冻结主界面。

#### FR-032 导出文本与文件结果
- **优先级**：P0
- **用户故事**：As a 运营或知识工作者, I want 一键复制或导出结果, so that 我能立即交付给外部对象。
- **描述**：系统应支持复制文本结果、导出文件到本地指定位置、打开原始文件位置。
- **业务规则**：
  1. 导出后必须提示保存位置；
  2. 文件命名需可识别；
  3. 导出状态与会话内展示状态区分。
- **边界条件**：
  - 导出路径不可写；
  - 文件重名；
  - 导出过程中磁盘空间不足。
- **验收标准**：
  1. 文本结果支持一键复制；
  2. 文件导出成功后有明确确认；
  3. 导出失败时给出失败原因与重试建议。

#### FR-033 代码、Diff 与结构化结果查看入口
- **优先级**：P1
- **用户故事**：As a 开发者, I want 更好地查看代码、diff 和 JSON 结果, so that 我能做专业判断。
- **描述**：系统应为代码、diff、JSON 等结构化结果提供增强查看入口。
- **业务规则**：
  1. MVP 可采用基础高亮或结构化树展示；
  2. Beta 继续扩展为更强 diff 浏览器；
  3. 导出前查看与最终文件内容一致。
- **边界条件**：
  - 超长 diff；
  - 混合编码；
  - 格式损坏。
- **验收标准**：
  1. JSON 可折叠浏览；
  2. 代码文本具备基础高亮；
  3. diff 至少可按新增/删除区分。

#### FR-034 日志与诊断包导出
- **优先级**：P1
- **用户故事**：As a 开发者或支持人员, I want 导出诊断信息, so that 我能排查问题或提交反馈。
- **描述**：系统应支持将关键日志、trace 信息、环境信息打包导出。
- **业务规则**：
  1. 诊断包需脱敏处理敏感字段；
  2. 导出内容至少包括：run trace、错误摘要、应用版本、Host 状态；
  3. 导出前提示包含内容范围。
- **边界条件**：
  - 日志过大；
  - 导出中断；
  - 某些日志文件不存在。
- **验收标准**：
  1. 用户能通过设置或错误页导出诊断包；
  2. 导出完成后可定位文件；
  3. 诊断包内容足以进行基础问题复现分析。

### 4.8 模块 H：长会话治理

#### FR-035 阶段性摘要生成
- **优先级**：P0
- **用户故事**：As an AI 重度用户, I want 长会话中自动生成阶段摘要, so that 我能快速继续工作而不是重新阅读全部历史。
- **描述**：系统需支持长会话阶段性摘要，服务于“继续执行”而非仅回顾。
- **业务规则**：
  1. 摘要记录生成时间点与覆盖范围；
  2. 摘要是上下文治理对象，不替代原始历史；
  3. 恢复时优先使用最新有效摘要。
- **边界条件**：
  - 摘要生成失败；
  - 摘要与实际上下文不一致；
  - 用户手动要求重新摘要。
- **验收标准**：
  1. 长会话达到阈值后自动出现摘要节点；
  2. 用户可查看摘要覆盖区间；
  3. 继续执行时可基于摘要进入。
- **[专家决策]**：摘要触发条件采用“消息量/工具事件量/上下文大小”综合阈值，而非仅按轮次。原因：更贴近 Cowork 类长任务产品的上下文治理方式。

#### FR-036 历史折叠与按需展开
- **优先级**：P0
- **用户故事**：As a 用户, I want 旧消息和长日志默认折叠, so that 当前重要信息更突出。
- **描述**：系统应折叠低价值长尾历史，并允许按需展开。
- **业务规则**：
  1. 折叠对象包括：旧消息、长工具卡片、长日志；
  2. 折叠必须保留可回看入口；
  3. 折叠不应改变原始数据。
- **边界条件**：
  - 关键审批消息被折叠；
  - 多层折叠嵌套；
  - 展开后性能下降。
- **验收标准**：
  1. 长消息流默认保持可读；
  2. 用户可一键展开具体块；
  3. 展开不丢内容。

#### FR-037 阶段标记与快速跳转
- **优先级**：P1
- **用户故事**：As a 产品/项目管理者, I want 看到会话被划分为若干阶段, so that 我能快速恢复心智模型。
- **描述**：系统需支持自动或手动阶段标记，并允许在阶段间快速跳转。
- **业务规则**：
  1. 阶段至少包含标题、时间范围、摘要；
  2. 用户可补充手动标记；
  3. 阶段跳转不改变数据，仅改变视图定位。
- **边界条件**：
  - 阶段过多；
  - 自动标记不准确；
  - 某阶段无摘要。
- **验收标准**：
  1. 长会话中可看到阶段节点；
  2. 点击节点可定位消息位置；
  3. 阶段标记与恢复流程联动。

#### FR-038 大会话性能降级策略
- **优先级**：P0
- **用户故事**：As a 用户, I want 即使会话很大也能顺畅操作, so that 产品不会越用越卡。
- **描述**：当会话规模过大时，系统需优先保证主界面可交互。
- **业务规则**：
  1. 采用懒加载、分段渲染、摘要替换、按需展开等策略；
  2. 降级必须可感知且可理解；
  3. 禁止将未加载误导为已丢失。
- **边界条件**：
  - 万级消息；
  - 大量工具卡片；
  - 恢复后首屏过载。
- **验收标准**：
  1. 大会话首屏渲染 ≤ 800ms；
  2. 降级时仍可滚动、切换和继续执行；
  3. 用户可主动加载更多历史。

### 4.9 模块 I：崩溃恢复与异常处理

#### FR-039 异常退出检测
- **优先级**：P0
- **用户故事**：As a 用户, I want 系统识别自己上次是否异常退出, so that 我知道是否需要恢复工作现场。
- **描述**：应用启动时应检测上次退出是否异常，并给出恢复提示。
- **业务规则**：
  1. 通过退出标记与最后状态快照判断是否异常；
  2. 若存在未完成 Run，应优先提示；
  3. 提示不得阻断用户放弃恢复。
- **边界条件**：
  - 正常关机但标记未写入；
  - 快照损坏；
  - 多个会话同时未完成。
- **验收标准**：
  1. 异常退出后下次启动能触发提示；
  2. 用户可选择恢复、放弃、查看日志；
  3. 错误恢复流程不会直接覆盖原数据。

#### FR-040 最近会话与未完成 Run 恢复
- **优先级**：P0
- **用户故事**：As an AI 重度用户, I want 重启后回到最近工作现场, so that 我不会丢掉半天进度。
- **描述**：系统需恢复最近会话、未完成 Run、待处理审批和有效上下文摘要。
- **业务规则**：
  1. 恢复优先级：未完成 Run > 最近会话 > 最近摘要；
  2. 恢复过程不自动重新执行危险动作；
  3. 用户必须能选择“继续 / 终止 / 仅查看”。
- **边界条件**：
  - Host 中断时正处于工具执行中；
  - 审批状态丢失；
  - 关联文件已不存在。
- **验收标准**：
  1. 崩溃重启后可恢复最近会话与未完成 Run；
  2. 待审批项仍能重新显示；
  3. 恢复后的状态与中断前关键事实一致。

#### FR-041 崩溃摘要展示
- **优先级**：P1
- **用户故事**：As a 开发者或高级用户, I want 知道上次崩溃发生了什么, so that 我能快速判断是否继续。
- **描述**：系统需展示上次崩溃摘要，包括时间、可能原因、影响范围。
- **业务规则**：
  1. 摘要需简洁可理解；
  2. 支持展开查看详细日志；
  3. 不应向普通用户暴露无法理解的原始堆栈作为唯一信息。
- **边界条件**：
  - 无日志；
  - 多次连续崩溃；
  - 仅 UI 崩溃但 Host 正常。
- **验收标准**：
  1. 崩溃提示至少包含时间和简要原因；
  2. 有查看详情入口；
  3. 用户可据此决定是否恢复或反馈。

#### FR-042 失败重试与人工接管
- **优先级**：P1
- **用户故事**：As a 用户, I want 在自动化失败时切回人工控制, so that 任务不会完全卡死。
- **描述**：当工具执行、预览、导出或自动化流程失败时，系统应提供重试或人工接管入口。
- **业务规则**：
  1. 失败上下文必须保留；
  2. 自动重试次数应有限；
  3. 人工接管路径需明确。
- **边界条件**：
  - 多次重试失败；
  - 外部依赖不可用；
  - 失败发生在 Full Auto 模式下。
- **验收标准**：
  1. 失败项可显式重试；
  2. 用户可选择切换回审批模式；
  3. 不因失败导致历史丢失。

### 4.10 模块 J：设置、诊断与版本兼容

#### FR-043 设置中心
- **优先级**：P0
- **用户故事**：As a 用户, I want 在一个地方配置模型、工作区和权限模式, so that 我能个性化系统行为。
- **描述**：系统应提供设置中心，至少包括模型参数、默认工作区、权限模式、日志与诊断入口。
- **业务规则**：
  1. 设置分为全局设置和会话默认值；
  2. 修改后需明确生效范围；
  3. 危险设置需二次确认。
- **边界条件**：
  - 工作区路径失效；
  - 模型配置不可用；
  - 配置文件损坏。
- **验收标准**：
  1. 设置修改后即时或下次会话生效；
  2. 生效范围明确；
  3. 配置异常时给出回退方案。

#### FR-044 日志查看与快速诊断入口
- **优先级**：P1
- **用户故事**：As a 开发者, I want 直接查看关键日志, so that 我能快速排查问题。
- **描述**：系统应在设置或错误页中提供日志查看与诊断入口。
- **业务规则**：
  1. 日志按时间倒序；
  2. 区分 UI、Host、运行日志；
  3. 支持复制和导出。
- **边界条件**：
  - 日志超大；
  - 某类日志缺失；
  - 权限不足。
- **验收标准**：
  1. 用户可查看最近日志摘要；
  2. 不同来源日志可区分；
  3. 从日志页可进入诊断包导出。

#### FR-045 OpenHarness 升级兼容适配
- **优先级**：P1
- **用户故事**：As a 产品维护者, I want OpenHarness 小版本升级不要求 UI 重构, so that 产品可持续迭代。
- **描述**：系统应通过 Host Adapter 隔离 OpenHarness 私有实现变化。
- **业务规则**：
  1. UI 仅依赖稳定协议和适配层；
  2. 上游升级引起的事件差异在适配层收敛；
  3. 升级失败可回退。
- **边界条件**：
  - 事件协议变化；
  - 存储结构变化；
  - 旧快照不兼容。
- **验收标准**：
  1. OpenHarness 小版本升级后 UI 主路径不需重构；
  2. 适配层有版本识别能力；
  3. 升级失败后可回退到上一个可用版本。
- **[专家决策]**：协议稳定性优先于直接暴露底层对象。原因：更符合长期桌面产品的可维护性要求。

#### FR-046 欢迎页与空状态引导
- **优先级**：P1
- **用户故事**：As a 首次用户, I want 在空白状态下知道下一步该做什么, so that 我能在 3 分钟内理解产品工作方式。
- **描述**：系统在首次启动、无会话、无消息、连接失败等空状态下，应提供明确引导。
- **业务规则**：
  1. 空状态说明应覆盖“创建会话”“添加材料”“发送任务”；
  2. 引导不得依赖技术术语；
  3. 有消息后欢迎页自动隐藏。
- **边界条件**：
  - 已有会话但当前为空；
  - Host 未连接；
  - 首次引导被跳过。
- **验收标准**：
  1. 首次用户可在 3 分钟内理解基本工作方式；
  2. 无消息时存在清晰空状态；
  3. 有消息后不干扰主工作流。

## 5. 非功能性需求

### 5.1 性能需求

| 编号 | 需求 | 指标 |
|---|---|---|
| NFR-001 | 冷启动时间 | ≤ 2.5 秒 |
| NFR-002 | Host ready 时间 | ≤ 4 秒 |
| NFR-003 | 会话切换响应 | ≤ 200ms |
| NFR-004 | 新建会话至可发送 | ≤ 1 秒 |
| NFR-005 | 审批 UI 响应 | ≤ 300ms |
| NFR-006 | 大会话首屏渲染 | ≤ 800ms |
| NFR-007 | 搜索结果返回 | ≤ 500ms |

### 5.2 稳定性需求
- NFR-008：Host 崩溃不能直接拖死 UI；
- NFR-009：UI 崩溃后重启可恢复最近工作上下文；
- NFR-010：关键写操作必须事务化；
- NFR-011：关键事件必须落盘；
- NFR-012：应用连续运行 8 小时主路径可用，不出现明显状态紊乱；
- NFR-013：发生局部功能失败时，主工作台仍需保持可交互。

### 5.3 可恢复性需求
- NFR-014：必须恢复最近 conversation / run / pending approvals；
- NFR-015：启动时检测异常退出标志；
- NFR-016：支持查看上次崩溃摘要；
- NFR-017：恢复不得自动重放高风险动作；
- NFR-018：恢复后的上下文必须优先可工作而非全量原始。

### 5.4 安全性需求
- NFR-019：默认不得启用 Full Auto；
- NFR-020：敏感操作必须审批或预授权；
- NFR-021：权限模型必须可见、可切换、可审计；
- NFR-022：工作区访问范围必须可控；
- NFR-023：日志与诊断包导出需进行敏感字段脱敏；
- NFR-024：本地数据需限定在用户可控目录，不得隐式上传云端；
- NFR-025：离线状态下不得伪装成功执行。

### 5.5 可用性需求
- NFR-026：非技术用户首次进入后应在 3 分钟内理解核心工作方式；
- NFR-027：系统状态、审批、错误、导出结果必须使用非底层术语表达；
- NFR-028：重要状态变化必须有显著视觉反馈；
- NFR-029：高风险模式需常驻提醒；
- NFR-030：主路径任务完成率达到可用水平（通过可用性测试评估，目标 ≥ 80%）。

### 5.6 兼容性需求
- NFR-031：UI 不直接依赖 OpenHarness 私有内部结构；
- NFR-032：OpenHarness 升级收敛在 Host Adapter；
- NFR-033：升级失败必须可回退；
- NFR-034：预览器失败时必须有降级视图；
- NFR-035：平台策略优先保证桌面体验一致性。

### 5.7 容量与治理需求
- NFR-036：本地数据库、日志、产物归档必须具备容量上限与清理策略；
- NFR-037：对超大文件、超长日志、超长会话需有明确提示与处理上限；
- NFR-038：超过容量边界时优先采用压缩、归档、折叠或手动清理方案；
- NFR-039：容量治理不得直接删除用户未确认的重要数据。

## 6. 数据需求

### 6.1 数据实体

| 实体 | 说明 | 关键字段 |
|---|---|---|
| Conversation | 会话主对象 | conversation_id, title, mode, workspace, status, created_at, updated_at |
| Message | 消息对象 | message_id, conversation_id, run_id, role, content, status, created_at |
| Run | 执行对象 | run_id, conversation_id, trace_id, status, started_at, ended_at |
| Approval | 审批对象 | approval_id, run_id, type, scope, status, created_at, decided_at |
| Artifact | 产物对象 | artifact_id, conversation_id, run_id, name, type, path, preview_status |
| ContextRef | 上下文引用对象 | ref_id, conversation_id, run_id, source_type, source_path/url, state |
| Summary | 阶段摘要对象 | summary_id, conversation_id, range_start, range_end, content, created_at |
| AuditEvent | 审计事件对象 | event_id, trace_id, event_type, payload, created_at |
| Snapshot | 快照对象 | snapshot_id, conversation_id, run_id, file_path, version |

### 6.2 存储策略
- 结构化数据存储于 SQLite；
- 兼容性/备份型快照存储为 JSON 文件；
- 产物、日志、预览缓存按目录归档；
- 索引数据与实际文件路径解耦，允许恢复或重新索引。

### 6.3 数据生命周期
1. 创建会话时生成 Conversation；
2. 提交任务时创建 Run，并附着 Message / ContextRef；
3. 执行过程中持续写入 Message、Approval、AuditEvent、Artifact；
4. 长会话中周期性生成 Summary；
5. 会话归档时保留数据但移出默认活跃视图；
6. 会话永久删除时清理结构化索引，并按策略删除/保留物理产物。

### 6.4 数据一致性要求
- Conversation、Message、Run 的主链路写入必须事务化；
- 关键状态变更需“先落盘后反馈”或采用等价一致性保障；
- SQLite 与 JSON 快照允许短暂最终一致，但必须可恢复对齐；
- 删除或归档必须记录审计日志。

### 6.5 备份与恢复
- 本地 JSON 快照作为恢复与兼容兜底；
- 恢复优先使用结构化库，如库损坏则可回退到快照恢复；
- 导出诊断包时可包含必要快照；
- 系统应支持重建索引而不要求原始数据全部丢弃。

### 6.6 已验证可行的数据能力
- SQLite 持久化；
- 会话与消息 CRUD；
- JSON 会话快照持久化；
- 历史恢复主链路。

## 7. 接口需求

### 7.1 外部接口

#### 7.1.1 Host HTTP 接口
最小 REST 接口用于：
- 创建会话；
- 获取会话列表；
- 恢复会话；
- 删除会话；
- 重命名会话；
- 健康检查。

**要求：**
- 接口返回稳定结构；
- 错误码与错误文案清晰；
- 不直接暴露 OpenHarness 私有内部对象。

#### 7.1.2 WebSocket 事件接口
WebSocket 作为主执行链路，负责：
- 提交任务；
- 接收流式文本；
- 接收工具事件；
- 接收审批请求；
- 接收运行完成与关闭事件。

**已验证可行事件映射：**
- `ready` → `session.ready`
- `AssistantTextDelta` → `assistant.delta`
- `AssistantTurnComplete` → `assistant.complete`
- `ToolExecutionStarted` → `tool.started`
- `ToolExecutionCompleted` → `tool.completed`
- `modal_request` → `approval.request` / `question.request`
- `line_complete` → `session.run_complete`
- `shutdown` → `session.closed`

### 7.2 内部接口

#### 7.2.1 UI 与 Host Adapter 接口
- UI 仅通过 Adapter 消费标准化协议；
- Adapter 屏蔽 OpenHarness 版本差异；
- Adapter 负责事件转换、状态归并、错误标准化。

#### 7.2.2 数据访问接口
- Repository 层负责 Conversation / Message / Run / Artifact 等读写；
- 数据层不得将底层 SQL 暴露到 UI；
- 支持按会话、按 Run、按 trace 查询。

### 7.3 用户界面约束
- 首发为单窗口工作台；
- 侧边栏 + 主会话区 + 输入区为默认布局；
- 审批请求需可在主流程中即时可见；
- Full Auto 需常驻标识；
- 基础预览器在应用内打开，不强制弹出外部应用；
- 欢迎页与空状态对首次用户友好，不使用底层工程术语。

### 7.4 协议兼容约束
- [专家决策] 前后端协议采用“有限稳定事件集合 + 扩展字段”策略，而非直接透传所有底层原始事件。原因：更有利于长期维护和向上兼容。

## 8. 约束条件

### 8.1 技术约束
1. 必须以 **import 方式集成 OpenHarness**，不得以 CLI 包装作为正式方案；
2. 桌面壳必须主导 product-state 数据；
3. UI 不可直接耦合 OpenHarness 私有实现；
4. 存储采用 SQLite + 本地归档 + JSON 快照双层模式；
5. 首期以单窗口工作台为主。

### 8.2 业务约束
1. 默认安全模式，不得默认 Full Auto；
2. 会话列表必须显示标题，不能显示 ID 作为主名称；
3. 用户必须理解 Safe / Balanced / Full Auto 差异；
4. 后端对普通用户不可见；
5. 长会话必须可治理、可恢复、可继续。

### 8.3 合规与安全约束
1. 本地文件、日志、诊断数据不得未经用户明确动作上传云端；
2. 审批记录与访问范围必须可审计；
3. 离线失败不得伪装成功；
4. 高风险动作必须可追踪责任链路；
5. 导出诊断包必须可控脱敏。

### 8.4 体验约束
1. 产品对标 Cowork 类“长时段协作工作台”，不是聊天机器人皮肤；
2. 以任务闭环为中心，而不是仅优化消息体验；
3. 以持续协作与恢复能力为核心差异化；
4. 用户不应被迫理解 CLI、进程树和日志结构。

## 9. 附录

### 9.1 状态机定义

#### 9.1.1 会话状态机

```text
[Draft/新建]
   │ 发送首条消息
   ▼
[Active/活跃]
   │ 归档
   ▼
[Archived/归档]
   │ 恢复
   ▼
[Active/活跃]
   │ 删除
   ▼
[Deleted/已删除]
```

**状态说明：**
- Draft：刚创建但尚未开始有效协作；
- Active：正常工作中的会话；
- Archived：已归档，不在默认活跃列表；
- Deleted：逻辑/物理删除后不可继续使用。

#### 9.1.2 连接状态机

```text
[Disconnected]
   │ 启动应用
   ▼
[Connecting]
   │ Host ready
   ▼
[Connected]
   │ 心跳异常/Host 崩溃
   ▼
[Degraded]
   │ 重试成功 ----------┐
   ▼                    │
[Disconnected] <--------┘
```

**业务规则：**
- Connecting 期间不可正式发起 Run；
- Degraded 表示部分能力可见但不可完整执行；
- Connected 才表示可正常工作。

#### 9.1.3 审批状态机

```text
[Pending]
  ├─ 用户批准一次 ──> [ApprovedOnce]
  ├─ 用户按作用域授权 -> [ApprovedScoped]
  ├─ 用户拒绝 -------> [Rejected]
  ├─ 超时/失效 -----> [Expired]
  └─ 会话终止 -------> [Cancelled]
```

#### 9.1.4 Run 状态机

```text
[Created]
   │ 提交执行
   ▼
[Running]
   ├─ 触发审批 -----> [WaitingApproval]
   │                    │ 审批通过
   │                    ▼
   │                 [Running]
   ├─ 用户停止 -----> [Stopped]
   ├─ 执行失败 -----> [Failed]
   ├─ 异常中断 -----> [Interrupted]
   └─ 正常完成 -----> [Completed]

[Interrupted]
   ├─ 用户恢复继续 -> [Running]
   ├─ 用户仅查看 --> [RecoveredViewOnly]
   └─ 用户终止 ----> [Stopped]
```

### 9.2 决策表

#### 9.2.1 权限模式决策表

| 场景 | Safe | Balanced | Full Auto |
|---|---|---|---|
| 普通读操作 | 自动执行 | 自动执行 | 自动执行 |
| 中风险文件操作 | 审批 | 视规则审批 | 自动执行 |
| 高风险操作 | 审批 | 审批/作用域授权 | 自动执行但需常驻提示 |
| 审计记录 | 必须 | 必须 | 必须 |
| 用户默认值 | 可默认 | 可默认 | 不可默认 |

#### 9.2.2 恢复策略决策表

| 条件 | 系统行为 |
|---|---|
| 上次正常退出 | 直接进入最近会话或默认首页 |
| 上次异常退出且有未完成 Run | 弹出恢复提示，允许继续/终止/仅查看 |
| 上次异常退出且快照损坏 | 显示崩溃摘要，建议进入诊断或回退到历史会话 |
| 有未处理审批 | 恢复后优先展示审批入口 |

### 9.3 边界用例

1. 会话中附加了本地文件，但该文件在执行前被外部删除；
2. URL 被识别为输入，但抓取失败；
3. 长会话存在上千条消息与大量工具卡片，恢复后仍需保证首屏可交互；
4. Full Auto 正在运行时 Host 崩溃，恢复后不得自动重放高风险动作；
5. 同一会话存在多个待审批项，用户仅处理其中一项；
6. 预览器不支持某文件格式时，必须降级显示元信息；
7. 删除会话时，部分产物文件已被手工移走；
8. OpenHarness 升级后新增事件字段，旧版 UI 仍应兼容忽略。

## 11. 结论

OpenHarness Desktop Electron 的首个可交付版本，必须不是“把 CLI 包进桌面壳”，而是一个真正的**桌面 AI 协作工作台**：
- 以会话为中心；
- 以 Run 为执行单位；
- 以审批和模式切换控制风险；
- 以产物、摘要、恢复支撑长时段协作；
- 以适配层隔离 OpenHarness 升级影响。

本文档定义的 SRS 基线可直接用于：
- 研发任务拆解；
- 测试用例设计；
- 版本范围控制；
- 设计与验收评审。
