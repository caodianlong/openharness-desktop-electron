# 产品需求文档 PRD v0.2

## 1. 产品定义

### 1.1 产品定位
**一句话定义：** OpenHarness Desktop 是一个面向通用知识工作场景的桌面 AI 协作助手，以桌面工作台方式承载搜索、阅读、写作、文件处理、数据分析与自动化执行，并以 OpenHarness 作为内嵌执行核心。

#### 用户价值定位
打造一个面向知识工作者的 **通用办公桌面 AI 协作助手**。产品体验对标 Claude Cowork 的“长时段协作工作台”，但不以编程为默认中心，而是覆盖搜索、阅读、写作、文件处理、数据分析、任务推进、自动化执行等办公全场景。

#### 技术定位
以 **OpenHarness** 为执行核心，采用 **import 方式深度集成**，而非依赖 CLI 进程包装。桌面应用负责工作台体验、状态可视化、审批与恢复能力，OpenHarness 负责 Agent 执行、工具编排与运行时上下文。

### 1.2 产品目标
优先实现以下用户价值型目标：
- 支持围绕同一任务进行 **长时段、跨阶段、可恢复** 的持续协作，而不是一次性问答
- 支持从输入、执行、查看结果到继续推进的 **工作流闭环**，减少用户在多个工具之间切换
- 提供 **可控自动化**：默认安全、过程可见、关键动作可审批，同时允许用户在明确授权下进入 Full Auto

在此基础上，产品应达成以下目标：
- 让用户以“桌面应用”的方式使用 OpenHarness，而不是命令行工具
- 提供完整的：
  - 会话管理
  - 会话历史
  - 任务运行
  - 审批控制
  - 文件/产物预览
  - 设置与恢复
- 默认安全模式，但必须支持 **Full Auto**
- 后端对用户不可见：
  - 一体启动
  - 一体退出
  - 无命令行黑窗口
- 保持 OpenHarness 可持续升级，桌面壳不与其私有实现强耦合

### 1.3 成功标准
#### 体验成功标准
- 非技术用户首次进入产品后，能够在 **3 分钟内理解基本工作方式**（会话、审批、结果查看、继续执行）
- 用户可在单一会话内完成“输入任务 → 执行 → 查看产物 → 继续迭代”的完整流程，**主路径任务完成率达到可用水平**
- 用户能够明确区分 Safe / Balanced / Full Auto 三种模式的差异与风险
- 用户能够在不理解底层 CLI、日志结构、进程细节的前提下完成主要操作
- 用户在长会话中仍能理解当前阶段、最近结果与下一步可执行动作

#### 产品与技术成功标准
- 冷启动进入主界面：**≤ 2.5 秒**
- 应用启动后 Host ready：**≤ 4 秒**
- 会话切换响应：**≤ 200ms**
- 新建会话到可发送第一条消息：**≤ 1 秒**
- 崩溃重启后可恢复最近会话与未完成 run
- OpenHarness 小版本升级不要求重构 UI 层

## 2. 用户画像与场景

### 2.1 核心用户画像

| 用户层级 | 用户类型 | 特征 | 核心诉求 |
|---|---|---|---|
| 通用知识工作者 | 内容、研究、分析类用户 | 高频处理资料、链接、文档、表格与总结任务，不关心底层技术实现 | 输入材料方便、结果可读、过程连续、可继续推进 |
| 组织协同角色 | 产品/项目管理者 | 需要围绕目标持续推进任务，关注状态、结论、交付物与风险控制 | 可跟进任务、看阶段进度、做审批、接续执行 |
| 业务执行角色 | 运营 | 高频进行信息收集、整理、文案、表格、复盘、流程执行 | 一站式完成搜索、整理、产出与导出 |
| 技术执行角色 | 开发者 | 会使用 AI 处理代码、文档、脚本、调试与自动化任务 | 稳定运行、可追溯、可恢复、可控自动化 |
| AI 重度用户 | 长会话/多阶段任务用户 | 一个任务持续数小时到数天，需要多轮迭代与上下文延续 | 历史管理、阶段摘要、恢复能力、Full Auto |

### 2.2 典型使用场景
#### 用户任务场景
1. **持续型项目会话**：围绕一个 workspace 或目标连续工作数天，要求历史、运行、产物完整保留。
2. **资料带入与分析**：将文件、链接、已有产物带入同一任务，要求系统理解上下文并继续工作。
3. **审批驱动的自动化执行**：默认模式下敏感动作需审批，支持按作用域授权。
4. **结果查看与交付**：在应用内查看文档、代码、diff、日志、图片、PDF，并可复制或导出给外部对象。
5. **异常恢复与继续执行**：Host 或 UI 崩溃后恢复最近会话、运行状态与待处理决策。

#### 系统能力场景
1. **长会话治理**：自动摘要、折叠历史、标记阶段节点，避免上下文膨胀影响可用性。
2. **Full Auto 模式**：用户显式开启后，在限定范围内自动执行，要求可见、可中断、可审计。
3. **审批与权限控制**：系统在触发敏感 action 时生成审批请求，并支持按范围授权。
4. **产物驱动的继续执行**：用户基于上一步生成的文件、日志、报告或代码继续发起下一步任务。
5. **运行追踪与诊断**：围绕 trace、工具调用、审批记录、错误信息进行排查与审计。

## 3. 核心问题与产品价值

### 3.1 当前问题
当前更高一层的问题不是“CLI 不好用”，而是 **现有 AI Agent 工具无法为通用知识工作者提供稳定、可持续、可控的桌面工作台体验**。

具体表现为：
- CLI 交互不等于桌面产品体验
- 进程与运行状态对普通用户不透明
- 会话、任务、审批、文件、日志分散
- 历史和恢复能力不足
- 若依赖 CLI 集成，上游升级会持续破坏产品层

### 3.2 产品价值
- **统一体验**：一个桌面应用，屏蔽底层复杂度。用户语言说明：用户只需要围绕任务工作，不需要理解底层命令、进程和工具拼装方式。
- **可控自动化**：默认安全，支持 Full Auto。用户语言说明：既能放心使用，也能在明确授权后把重复动作交给系统自动完成。
- **可追溯**：会话、审批、产物、日志、trace 全链路。用户语言说明：每一步发生了什么、生成了什么、为什么中断，都能回看和解释。
- **可恢复**：崩溃后可回到工作现场。用户语言说明：不会因为窗口关闭、程序崩溃或重启，就把半天工作全部丢掉。
- **可升级**：OpenHarness 升级收敛在适配层。用户语言说明：产品可以持续迭代，不会因为底层升级频繁破坏用户体验。

## 4. 功能范围

> 本文所述 MVP 指"首个可交付的产品版本边界"，不同于当前研发团队已完成的关键技术验证范围。

### 4.1 MVP

#### 必须包含
- 桌面应用壳
- 会话列表：新建、重命名、归档、删除
- 会话消息流
- 基础 run / task 状态展示
- 审批弹窗与审批队列
- 文件/产物列表
- 文件带入会话
- 任务上下文摘要
- 结果导出
- 基础预览器：文本、Markdown、图片、PDF、JSON
- 设置中心：模型参数、默认工作区、权限模式、日志与诊断
- 本地数据库（SQLite）
- 本地归档（artifacts / logs / snapshots）
- Host 启停与健康检查
- 崩溃恢复
- Safe / Balanced / Full Auto 模式切换

#### 明确不做
- 多人协作
- 云同步
- 插件市场
- 企业后台
- 暂不支持复杂多窗口协同，首期以内嵌单窗口工作台为主
- 重写 OpenHarness 内核

### 4.2 Beta
**版本目标：** 在 MVP 可交付基础上，补齐高频使用中的效率能力与管理能力，让产品从“能用”进入“适合持续使用”。

- 任务中心
- 工作区管理
- 审批规则模板
- 更丰富预览器（diff、代码高亮、HTML sandbox）
- trace 浏览器
- 会话搜索
- 历史筛选与标签
- OpenHarness 版本管理
- 增强恢复（run 级恢复）

### 4.3 1.0
**版本目标：** 在稳定主路径基础上，完善组织级使用、复杂任务治理与产品级交付能力，使其成为可长期依赖的桌面 AI 工作台。

- 多项目工作区
- 细粒度权限策略
- 任务看板
- 批量审批
- 完整诊断报告
- 自动更新与回滚
- 企业策略注入预留
- 远程执行扩展预留

## 5. 非功能需求

### 5.1 性能
**用户意义：** 产品必须像本地办公软件一样足够快，不能让等待本身成为使用阻力。

- 冷启动：≤ 2.5 秒
- Host ready：≤ 4 秒
- 会话切换：≤ 200ms
- 审批 UI 响应：≤ 300ms
- 大会话首屏渲染：≤ 800ms

### 5.2 稳定性
**用户意义：** 用户应能放心把长任务交给系统，而不是担心程序异常导致界面卡死或结果丢失。

- Host 崩溃不能直接拖死 UI
- UI 崩溃后重启可恢复最近工作上下文
- 关键写操作必须事务化
- 关键事件必须落盘

### 5.3 可恢复性
**用户意义：** 当系统中断时，用户能快速回到刚才的工作阶段，而不是重新组织上下文和操作步骤。

- 恢复最近 conversation / run / pending approvals
- 检测异常退出标志
- 支持查看上次崩溃摘要

### 5.4 权限与安全
**用户意义：** 用户需要在“效率”和“风险”之间有明确控制权，知道系统能做什么、何时需要自己确认。

- 默认非 Full Auto
- 敏感操作需审批或预授权
- 权限模型必须可见、可切换、可审计
- 访问范围可控
- 审批记录可追溯

### 5.5 诊断与追溯
**用户意义：** 当结果异常、执行失败或需要复盘时，用户和研发都能快速定位问题来源。

- 每个 run 有 trace_id
- 工具调用、审批、产物、错误全链路关联
- 支持导出诊断包

### 5.6 升级兼容
**用户意义：** 产品升级不应破坏已形成的工作习惯，也不应因底层变化频繁要求前端重做。

- UI 不直接依赖 OpenHarness 私有内部结构
- OpenHarness 升级收敛在 Host Adapter
- 升级失败必须可回退

## 6. 关键交互说明

### 6.1 首次启动
1. 用户启动桌面应用
2. Electron Main 拉起 Python Headless Host
3. UI 显示启动状态
4. Host ready 后进入主界面
5. 首次使用引导设置工作区、模型、权限模式

### 6.2 新建会话
1. 用户点击“新建会话”
2. 选择工作区与权限模式
3. 创建 conversation
4. 初始化 runtime context
5. 进入主工作区界面

### 6.3 执行与审批
1. 用户发送任务
2. OpenHarness 运行
3. 如触发敏感 action，生成 approval request
4. UI 弹出审批卡片或放入审批队列
5. 用户批准/拒绝/按作用域授权
6. Host 接收决策后继续执行

### 6.4 Full Auto 切换
1. 用户显式开启
2. 系统展示风险说明与作用域
3. 记录审计事件
4. 顶部常驻 Full Auto 标识
5. 支持随时关闭并回退为审批模式

### 6.5 崩溃恢复
1. 启动时检测异常退出
2. 恢复最近快照
3. 标记未完成 run
4. 用户可继续/终止/查看日志

### 6.6 文件加入任务
1. 用户在当前会话中选择本地文件、拖拽文件或引用已有产物
2. 系统将文件登记为当前任务输入，并展示文件类型、名称、大小与可预览状态
3. 系统将文件与当前 conversation / run 建立关联
4. 用户可在发送前补充说明“如何使用该文件”
5. OpenHarness 在运行时读取文件引用并纳入上下文
6. 任务完成后，输入文件与输出产物可形成可追溯链路

### 6.7 从产物继续执行
1. 用户在结果区、产物列表或预览器中选择一个已有产物
2. 用户点击“基于此继续”或类似入口
3. 系统自动生成带产物引用的下一步任务上下文
4. 用户补充目标或修改指令
5. 新 run 在保留原上下文关系的前提下继续执行
6. 产物链路在历史中可回看、可跳转、可追溯

## 9. 信息输入与上下文管理

### 9.1 目标
建立统一的信息输入模型，使用户可以自然地把任务相关材料带入会话，并在长周期内保持上下文可理解、可追溯、可治理。

### 9.2 输入来源
产品首期至少支持以下输入来源：
- 用户直接输入文本
- 本地文件加入会话
- URL 粘贴
- 工作区文件引用
- 历史产物引用
- 系统恢复时带回的上下文摘要

### 9.3 文件输入要求
- 支持通过选择器、拖拽、粘贴等方式将文件加入当前会话
- 文件进入会话后需显示基础元信息：文件名、类型、大小、来源
- 系统需区分“仅引用路径”和“已纳入任务上下文”的状态
- 对可预览文件类型，应支持直接预览
- 文件与 conversation / run 的关系需可追踪

### 9.4 URL 粘贴要求
- 用户粘贴 URL 后，系统应识别其为可处理输入，而非普通文本噪声
- 系统应展示该 URL 的引用状态，并允许用户说明用途
- 对支持抓取或解析的 URL，系统需在执行前明确展示是否已纳入上下文
- URL 解析失败时，需给出可理解提示和降级方案

### 9.5 工作区引用要求
- 用户可从当前工作区中引用已有文件或目录作为任务上下文
- 系统应明确展示引用范围，避免用户不清楚哪些内容会被访问
- 引用需受当前权限模式约束，并纳入审计范围

### 9.6 上下文生命周期
- 会话上下文需区分：即时输入、阶段摘要、历史折叠内容、持久引用对象
- 长会话中，系统需支持自动摘要与阶段收敛，避免无限膨胀
- 被折叠或摘要的信息不能等同于删除，应可回看或重新展开
- 恢复后的会话应优先展示“当前有效上下文”而不是全部原始细节

## 10. 结果导出与交付

### 10.1 目标
让用户能够将系统输出自然转化为可复用、可分享、可交付的工作成果，而不是停留在聊天窗口内部。

### 10.2 基础导出能力
MVP 至少支持以下结果处理方式：
- 一键复制文本结果
- 导出文件到本地指定位置
- 从产物列表打开原始文件位置
- 对日志、诊断包、结构化结果提供单独导出入口

### 10.3 导出对象范围
- 文本回答
- Markdown / 文档类产物
- 代码与 diff
- 图片与 PDF
- JSON / 结构化数据
- 日志与诊断包

### 10.4 交付要求
- 导出动作需明确成功状态与输出位置
- 导出后的文件命名应具有可识别性，避免大量匿名临时文件
- 用户应能区分“会话内展示结果”和“已导出交付结果”
- 对未来外部集成（如第三方文档、IM、任务系统）预留接口，但不纳入首期交付范围

## 11. 长会话治理

### 11.1 目标
确保单个任务持续数小时到数天时，系统仍保持可读、可控、可恢复，而不会因信息堆积导致体验崩坏。

### 11.2 摘要机制
- 系统应支持对长会话生成阶段性摘要
- 摘要需服务于“继续执行”，而非仅服务于回顾
- 用户应可看到摘要生成的时间点与覆盖范围

### 11.3 折叠机制
- 历史消息、工具卡片、长日志支持折叠
- 默认应优先突出当前有效信息，弱化低价值长尾细节
- 折叠不等于丢失，用户可手动展开查看

### 11.4 阶段标记
- 支持将长会话按任务阶段进行标记
- 阶段标记可由系统自动生成，也可由用户手动补充
- 阶段间应支持快速跳转，帮助用户恢复工作心智模型

### 11.5 性能降级策略
- 当会话规模过大时，系统应优先保证主界面可交互
- 可通过懒加载、摘要替换、按需展开、分段渲染等方式降级
- 降级必须可感知且可理解，不能让用户误以为数据丢失

## 12. 平台约束与降级策略

### 12.1 首发平台范围
- 首期以桌面端为唯一正式交付平台
- 首发形态为单机本地应用，不依赖云端账户体系作为基础前提
- 平台策略优先保证核心体验一致性，而非过早追求多端同步

### 12.2 容量边界
- 本地数据库、日志、产物归档需设置可治理上限与清理策略
- 对超大文件、超长日志、超长会话需有明确提示与处理上限
- 当超过容量边界时，应优先提供压缩、归档、折叠或手动清理方案

### 12.3 离线行为
- 在本地 Host 可运行的前提下，应用应尽可能维持本地历史、产物查看、设置修改等基础能力
- 对必须联网的能力，应清晰区分“不可用”与“暂时失败”
- 离线时不得伪装成已执行成功

### 12.4 失败降级
- Host 未就绪时，UI 需提供明确状态，而不是空白界面
- 某类预览器失败时，应退回到基础文件信息或原始内容视图
- 工具执行失败时，应保留失败上下文、错误信息与可重试入口
- 自动化执行失败时，应允许用户切回人工审批或手动继续
- OpenHarness 升级失败时，应支持回退到上一可用版本
