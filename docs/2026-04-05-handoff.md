## 3. 关键上下文

### 项目定位
- **对标 Claude Cowork**（通用办公协作助手），不是 Claude Desktop（编程专用）
- 支持搜索、阅读、写作、文件处理、分析、编程等全场景
- 编程是能力之一，不是默认中心用户

### 用户明确要求
- 用 Gemini 编码，助手负责规划 + 验收
- 每次只开发一个功能，验收通过后再继续下一个
- 以 `import` 方式调用 OpenHarness（非 CLI）
- 会话列表必须显示标题，不能显示 ID
- 后端同时充当前端 HTTP Server（解决跨域，无需单独 serve 前端）

### 架构决策
| 决策点 | 选择 | 原因 |
|---|---|---|
| 集成方式 | import 库 | 非 CLI |
| 后端同源 serve | FastAPI 同时 serve 前端 index.html | 解决跨域，简化部署 |
| 端口 | 8789（后端 + 前端同源） | 单一入口 |
| REST API | 统一 `/api/...` 前缀 | 路径规范 |
| WebSocket | `/ws/{session_id}` 不变 | 与 OpenHarness 事件兼容 |
| 存储 | 双层 SQLite + JSON 快照 | 查询性能 + 升级兼容 |
| 编码方式 | Gemini CLI，助手规划+验收 | 效率 |

### 运行环境
- 后端：`apps/host-python/.venv/bin/python3 -m uvicorn host_mvp.ws_server:app --host 0.0.0.0 --port 8789`
- 必须用 `.venv/bin/python3`（websockets 13.1 不兼容系统 Python 3.12）
- `PYTHONPATH=src` 环境变量
- 健康检查：`curl -s http://127.0.0.1:8789/api/health`
- 系统 exec 后台进程容易被 SIGKILL，用 `nohup` 或 `init.sh` 启动

### 重要路径
- 项目根目录：`~/dev/openharness-desktop-electron/`
- 后端模块：`apps/host-python/src/host_mvp/`
- 前端：`apps/host-python/frontend/index.html`
- SQLite：`~/.openharness/desktop.db`
- 后端日志：`/tmp/backend.log`
- vendor：`vendor/OpenHarness/src`
- 参考文档：`docs/refs/`（Coding_Agent_需求分析师.md、技术架构师.md、harness-design-long-running-apps.md）

## 4. 关键发现

### 核心事件映射（已验证全链路）
```
OpenHarness 原生          →  WS 事件
──────────────────────────────────────────
ready                     →  session.ready
AssistantTextDelta        →  assistant.delta
AssistantTurnComplete     →  assistant.complete
ToolExecutionStarted      →  tool.started
ToolExecutionCompleted    →  tool.completed
transcript_item           →  transcript.item
modal_request             →  approval.request / question.request
line_complete             →  session.run_complete
shutdown                  →  session.closed
```

### 全链路已验证通过
- 发消息 → agent 执行 → 流式输出 → 工具调用 → 结果渲染 → SQLite 保存 → 历史恢复
- `/help` slash 命令 → OpenHarness CommandRegistry → 返回完整命令列表
- `/model show` → 返回当前模型名

### 数据库表结构
现有表：`sessions`、`messages`、`approvals`、`artifacts`
新增列（通过 ALTER TABLE 自动添加）：
- `sessions.permission_mode` TEXT DEFAULT 'full_auto'
- `sessions.was_running_runtime` INTEGER DEFAULT 0

### REST API 全景
- `GET /api/health` — 健康检查
- `POST /api/sessions` — 创建会话
- `GET /api/sessions` — 会话列表
- `GET /api/sessions/{id}` — 会话详情 + 消息
- `POST /api/sessions/{id}/resume` — 恢复会话
- `POST /api/sessions/{id}/fork` — 分叉会话
- `PUT /api/sessions/{id}/title` — 重命名
- `PUT /api/sessions/{id}/permission` — 切换权限模式
- `DELETE /api/sessions/{id}` — 删除
- `GET /api/sessions/{id}/approvals` — 审批列表
- `GET /api/sessions/{id}/artifacts` — 产物列表
- `GET /api/artifacts/{id}` — 产物详情
- `GET /api/artifacts/{id}/preview` — 产物预览
- `GET /api/settings` — 获取配置
- `PUT /api/settings` — 保存配置
- `GET /api/logs` — 后端日志（最近 500 行）
- `GET /api/recovery` — 异常中断会话列表
- `POST /api/sessions/{id}/resume` — 恢复（含 was_running_runtime 清零）

## 7. 风险与注意事项

### ⚠️ 易踩的坑
1. **session_store.py 的 from_row() 列映射** — 通过 ALTER TABLE 加了 `permission_mode` 列后，row 索引需要按新旧兼容方式读取。已在 session_store.py 中做了动态列判断，但如果还有 ALTER TABLE 增加列，需要再次检查。
2. **系统 SIGKILL** — exec 后台进程超时后被杀，后端不活了就用 `init.sh` 重启。
3. **`.venv/bin/python3`** — 必须用 venv 的 Python，系统 Python 3.12 与 websockets 13.1 不兼容。
4. **`Path(__file__).parents[4]`** — ws_server.py 中定位项目根目录的层级，新增/移动文件时注意。
5. **后端重启需要时间** — 启动后等 3 秒再验证，轮询太早会返回空。
6. **`was_running_runtime` 列** — 通过 ALTER TABLE 自动添加，首次运行可能需要手动执行 `ALTER TABLE sessions ADD COLUMN was_running_runtime INTEGER DEFAULT 0;`
7. **`approvals` 表和 `artifacts` 表** — 在 session_store.py 的 `_ensure_db()` 中通过 executescript 的 CREATE TABLE IF NOT EXISTS 自动初始化。
8. **`session_store.py` 的 `update_session()` 不支持 `was_running_runtime`** — ws_server.py 中直接用 SQLite 语句更新该列。

### ✅ 已验证通过的方向（不建议改）
- OpenHarness 事件映射（完全复用原生类型）
- SQLite + JSON 双层存储
- `import` 调用 OpenHarness
- REST `/api/...` 前缀模式
- WebSocket `/ws/{session_id}` 协议
- 后端同源 serve 前端（GET / 返回 index.html，catch-all 最后注册避免抢路由）
